# 渠道字段全量模拟编辑设计稿

- 日期：2026-04-02
- 主题：Channel Field Editing for Omni-Channel Product Center
- 项目：`next-shadcn-admin-dashboard`
- 目标：在现有商品详情页的渠道 Tab 中，加入一套可编辑、可保存、可联动的全量模拟字段编辑能力，用页面保存行为反向逼出未来真实接口的写入与状态重算边界。

## 1. 背景

当前“全渠道商品中心”第一阶段已经完成以下基础能力：

1. 商品中心首页，支持摘要卡、筛选区、商品列表。
2. 商品详情页，支持主档信息、SKU、素材状态、渠道发布状态。
3. 发布工作台，支持基于渠道状态的任务汇总与跳转。

当前缺口在于：详情页中的渠道发布区域仍然是只读的，用户只能看“缺失字段 / 渠道专属字段 / 状态”，不能直接补齐字段并观察状态变化。这使得页面虽然具备信息架构，但还没有真正进入“页面数据从哪来、保存后状态怎么变”的闭环。

本阶段目标是在不接真实平台 API 的前提下，把渠道字段编辑、保存反馈、跨页面联动模拟出来。

## 2. 已确认决策

### 2.1 编辑形态

- 编辑发生在商品详情页的 `抖音 / 视频号` Tab 内。
- 采用 `按渠道分 Tab 内联编辑` 方案，不使用右侧抽屉或独立编辑模式切换。
- 每个渠道 Tab 下同时展示状态摘要与字段编辑区。

### 2.2 保存行为

- 字段修改先进入本地表单状态。
- 用户显式点击 `保存字段` 后才生效。
- 不做自动保存。

### 2.3 字段范围

本阶段覆盖四类字段：

1. 文本字段
2. 选择字段
3. 媒体字段，但只允许从当前商品已有素材中选择
4. 数组字段

### 2.4 联动范围

保存成功后，状态需要同步反映到：

1. 当前详情页
2. 商品中心首页
3. 发布工作台

### 2.5 范围边界

本阶段明确不做：

- 真实上传
- 真实平台提交审核 / 上架动作
- 自动保存
- 数组拖拽排序
- 独立素材中心
- 后端 API 接入

## 3. 信息架构

渠道 Tab 中的内容固定分成三段。

### 3.1 状态摘要区

位于最上方，只读，不进入编辑。

展示内容：

- 发布状态
- 审核状态
- 上下架状态
- 最近同步时间
- 当前缺失字段数量

职责：

- 告诉用户当前卡在哪
- 保存成功后即时反映状态变化

### 3.2 字段分组编辑区

位于中间，是本阶段的主体。

按业务语义拆成四组：

1. `基础发布信息`
   - 渠道短标题
   - 渠道类目
   - 卖点摘要

2. `履约与配送`
   - 运费模板
   - 发货时效
   - 服务承诺

3. `内容素材`
   - 封面图
   - 轮播图
   - 主图视频
   - 素材仅能从当前商品已有 `assets` 中选择

4. `扩展数组字段`
   - 卖点标签
   - FAQ
   - 渠道亮点
   - 补充说明

### 3.3 保存反馈区

位于底部。

包含：

- 主按钮 `保存字段`
- 当前是否有未保存变更
- 最近一次保存时间
- 保存后状态变化摘要，例如：
  - 缺失字段从 3 项降为 1 项
  - 渠道状态从 `missing_fields` 变为 `ready_to_list`

## 4. 字段交互设计

### 4.1 文本字段

控件：

- `Input`
- `Textarea`

要求：

- 支持必填标识
- 支持占位文案
- 支持字段级错误提示

示例校验：

- 短标题不能为空
- 卖点摘要字数不能超出配置上限

### 4.2 选择字段

控件：

- `Select`
- 必要时可升级为 `Combobox`

要求：

- 选项来自本地 mock options
- 允许明显的“未选择”态
- “未选择”直接参与缺失字段判断

### 4.3 媒体字段

控件模式：

- 当前已选素材预览
- “从素材中选择”入口
- 打开轻量素材选择层
- 仅展示当前商品已有的 `assets`

要求：

- 不做真实上传
- 不做跨商品素材复用
- 仅模拟“选择后回填字段”

### 4.4 数组字段

控件模式：

- `新增一项`
- `删除一项`
- 每项由简单输入块组成

要求：

- 第一版不做拖拽排序
- 保持线性增删即可

## 5. 保存与联动数据流

本阶段最关键的是把“保存字段”视为真正的写操作，而不是单页局部状态更新。

### 5.1 保存前

- 表单值先存在渠道 Tab 的本地表单状态
- 页面提示是否存在未保存变更
- 首页与工作台状态不变化

### 5.2 保存时

点击 `保存字段` 后执行：

1. 字段校验
2. 生成标准化的渠道字段 payload
3. 写入共享 mock 数据源
4. 重算当前渠道的缺失字段
5. 重算当前渠道的发布状态
6. 更新时间戳与保存反馈

### 5.3 保存后联动

保存成功后，必须同时影响三处视图：

1. `详情页`
   - 缺失字段
   - 发布状态
   - 状态摘要
   - 最近同步时间

2. `首页`
   - 商品资料完整度
   - 抖音状态
   - 视频号状态

3. `发布工作台`
   - 任务列表
   - 顶部概览卡片

## 6. 前端模块边界

本阶段不建议继续把数据当成只读常量直接消费，而要升级成轻量共享数据层。

### 6.1 字段配置模块

职责：

- 定义每个渠道的字段组
- 定义字段类型
- 定义字段标签、占位文案、校验规则、默认值

建议抽象：

- `ChannelFieldGroupConfig`
- `ChannelFieldConfig`

### 6.2 渠道表单模块

职责：

- 渲染四类字段
- 管理脏状态
- 管理校验结果
- 触发保存

### 6.3 素材选择模块

职责：

- 在当前商品素材集合中提供选择能力
- 回填媒体字段值

### 6.4 共享 mock 数据层

职责：

- 提供商品详情、首页列表、工作台三处共用的读数据
- 提供保存字段后的写能力
- 负责状态重算

### 6.5 状态重算模块

职责：

- 根据字段完整度重算 `missing_fields / ready_to_list / rejected / sync_error` 等状态
- 输出首页和工作台需要的衍生数据

## 7. 未来真实接口边界

这个阶段的保存行为会自然倒推出未来真实系统至少需要两类写能力：

1. `保存渠道字段`
   - 输入：某个商品、某个渠道、一组字段值
   - 输出：保存后的字段结果

2. `重算渠道状态 / 工作台任务`
   - 输入：保存后的渠道字段
   - 输出：新的状态摘要、缺失字段、工作台任务

也就是说，这一阶段不是在“模拟页面”，而是在先行固化未来接口的输入输出结构。

## 8. 范围

### 8.1 In Scope

- 详情页渠道 Tab 内联编辑
- 四类字段渲染
- 显式保存字段
- 表单校验
- 保存结果反馈
- 详情页 / 首页 / 工作台三处联动
- 基于字段配置的表单分组渲染

### 8.2 Out of Scope

- 真实媒体上传
- 真实平台 API
- 自动保存
- 数组排序
- 批量编辑
- 审核 / 上架真实动作

## 9. 完成标准

本阶段完成后，应满足以下标准：

1. 用户能在详情页渠道 Tab 中编辑文本、选择、媒体、数组四类字段。
2. 用户点击 `保存字段` 后能看到明确的校验与保存反馈。
3. 保存后当前渠道的缺失字段与状态会变化。
4. 首页商品列表会反映新的渠道状态。
5. 发布工作台会反映新的任务数量和任务状态。
6. 字段结构已经足够清晰，后续可以替换成真实接口而不用推翻页面交互。

## 10. 风险与后续演进

### 10.1 当前风险

- 如果字段全部硬编码在组件里，后续接更多平台会迅速失控。
- 如果保存只更新详情页局部状态，首页和工作台会与真实心智脱节。
- 如果媒体选择逻辑提前做成上传器，会把这一阶段拖进素材中心建设。

### 10.2 后续演进方向

- 把字段配置提炼成更完整的 renderer 体系
- 接入真实素材中心
- 接入真实渠道字段保存接口
- 在保存字段之后接入真实审核 / 上架动作

---

结论：本阶段应把渠道字段编辑做成“分组表单 + 一次保存 + 三处联动”的完整模拟闭环。这样既能覆盖全量字段类型，又能用最小复杂度逼出未来真实接口和状态流的边界。
