# Douyin Dynamic Platform Panel Design

## 背景

当前统一发布页已经具备：

- `通用商品区 + 平台扩展区 + 右侧发布控制区` 的整体骨架
- 视频号平台区的真实编辑能力
- 发布页右侧基于共享 store 的动作区

但抖店平台区仍然停留在静态说明卡层：

- 类目是静态文案
- 属性区不是动态表单
- 规则区不是类目驱动结果
- 右侧动作区无法基于抖店类目/属性状态给出可信的 mock 判断

这会让统一发布页在抖店场景下只有结构，没有真正的“平台扩展层”能力。

## 目标

在不接真实抖店 API 的前提下，把抖店平台区升级成一个最小可交互的动态扩展区：

1. 类目成为抖店平台区的驱动源
2. 类目变化时，属性区和规则区一起变化
3. 右侧发布控制区能对抖店当前动态状态做出响应
4. 整体仍然保持“通用层 + 平台扩展层”的架构，不让抖店反向污染通用区

## 非目标

这一轮明确不做：

- 真实 `/product/getCatePropertyV2`
- 真实 `/product/getProductUpdateRule`
- 真实抖店发布/编辑接口调用
- 完整品牌库和资质上传
- 完整抖店 SKU 属性矩阵
- 通用区跟随抖店类目重排

## 设计原则

### 1. 通用区不动

商品基础、通用素材、SKU 基础、默认履约仍然属于平台无关层。  
抖店这轮的动态能力只能活在 `平台配置 -> 抖店` 区域内。

### 2. 类目是唯一驱动源

抖店平台区的动态变化统一由“当前选中的抖店类目”触发，不允许每个区块自行维护一套状态。

### 3. 动态只做最小闭环

这轮只验证：

- 类目切换
- 属性 schema 切换
- 规则摘要切换
- 右侧动作提示切换

不追求完整的 add/edit DTO 或真实 API 能力。

### 4. 平台差异只下沉到平台区

抖店专属的字段、规则、资质提醒、图文限制，都应保持在 `douyin extension` 中，不进入通用模型。

## 页面信息架构

统一发布页仍保持三栏逻辑：

1. 左侧上半：`通用商品区`
2. 左侧下半：`抖店平台区`
3. 右侧：`发布校验 + 发布控制区`

### 抖店平台区的新结构

抖店 Tab 升级为 4 段：

1. `类目选择`
2. `类目属性区`
3. `规则摘要区`
4. `平台状态提示`

#### 类目选择

用户可在一组 mock 类目中切换，例如：

- 咖啡冲饮 > 挂耳咖啡
- 茶饮冲调 > 浓缩茶饮
- 零食礼盒 > 伴手礼盒

类目切换后，抖店平台区内的动态内容立即刷新。

#### 类目属性区

按当前类目渲染最小属性表单。第一轮只覆盖两类：

- 销售属性：1-2 项
- 关键非销售属性：2-4 项

每项都带：

- 字段标签
- 必填标识
- 当前值 / 未填态

这一层的目标是验证“类目驱动 schema”，不是覆盖全量属性。

#### 规则摘要区

基于当前类目与属性完成度，动态显示：

- 必填属性规则
- 品牌/资质提醒
- 图文限制摘要

规则区不等于字段区，它负责告诉用户：

- 当前还差什么
- 哪些限制会阻断后续动作

#### 平台状态提示

保留一个轻量的抖店平台状态块，提示：

- 当前类目是否已选
- 当前属性是否齐备
- 当前规则是否有阻塞项

它服务右侧动作区，但不重复右侧完整动作按钮。

## 动态数据模型

建议新增一层纯前端 mock 模型，而不是直接把字段散进组件里。

### DouyinCategoryOption

- `id`
- `label`
- `path`

### DouyinDynamicField

- `fieldId`
- `label`
- `type`
- `required`
- `group`
- `options?`
- `placeholder?`

### DouyinRuleSummary

- `requiredHighlights`
- `brandHint`
- `qualificationHint`
- `mediaHint`
- `blockers`

### DouyinDynamicPanelState

- `selectedCategoryId`
- `fields`
- `values`
- `rules`
- `missingFields`

这层状态只服务抖店平台区和右侧发布控制区。

## 动态行为

### 类目切换

切换类目后：

1. 抖店属性字段组刷新
2. 规则摘要刷新
3. 抖店缺失字段重新计算
4. 右侧发布控制区如果当前平台是抖店，则主动作和提示重新推导

### 属性编辑

第一轮只要求：

- 可编辑
- 必填字段有缺失态
- 完成度可计算

保存逻辑先不接共享 store 写回商品主档，可先保持在抖店平台区本地状态内。  
但右侧发布控制区要读取这份实时状态。

### 右侧动作联动

当当前平台是抖店时，右侧动作区至少能区分三种状态：

1. `未选类目 / 属性缺失`
主动作：`继续补字段`

2. `字段齐备但仍有规则阻塞`
主动作：`更新渠道`
提示：显示当前阻塞项

3. `字段齐备且无显式阻塞`
主动作：`提交审核` 或 `更新渠道`

这里仍然是 mock 动作，不触发真实 API。

## 组件拆分建议

### 新增

- `douyin-category-selector.tsx`
- `douyin-dynamic-attribute-section.tsx`
- `douyin-rule-summary.tsx`
- `douyin-platform-panel.tsx`

### 调整

- `platform-publish-panel.tsx`
  - 当前静态 `DouyinCategorySection / DouyinRuleSection` 改为真实抖店平台面板入口

- `publish-validation-summary.tsx`
  - 当当前平台是抖店时，读取抖店动态状态，而不只是静态平台 metadata

- `publish-action-panel.tsx`
  - 为抖店增加一条“基于动态平台状态”的动作推导入口

## 状态边界

为了避免把抖店动态状态过早塞进全局 store，这一轮建议采用：

- 发布页局部 `DouyinPanelState`
- 右侧动作区通过 props 或 context 读取当前抖店动态状态

而不是：

- 直接改 `product-center store`
- 直接把抖店动态属性写回通用商品主档

原因：

1. 这轮目标是验证平台动态区结构
2. 真实抖店类目/规则接口还没接
3. 过早把 mock 动态状态写入全局 store，会加重未来契约迁移成本

## 完成标准

1. 切到抖店平台区时，类目可以切换，不再是静态说明卡
2. 类目变化时，属性区和规则区一起变化
3. 当前平台是抖店时，右侧动作区能对类目/属性完成度和规则阻塞做出真实响应
4. 通用区结构不变，平台差异仍然严格下沉在抖店扩展区
5. 代码边界清楚，为后续接 `/getCatePropertyV2` 和 `/getProductUpdateRule` 留出自然入口

## 风险与控制

### 风险 1：抖店状态提前污染全局 store

控制：
本轮抖店动态状态优先留在发布页局部，不直接写回 `product-center store`。

### 风险 2：类目一改，整个页面联动过重

控制：
只让抖店平台区和右侧动作区联动，不让通用区跟着重排。

### 风险 3：动态 schema 做过重

控制：
先只做最小字段组和规则摘要，不碰全量类目属性和复杂 SKU 规则。

### 风险 4：右侧动作区重新走成静态文案

控制：
明确要求右侧动作区读取抖店动态状态，而不是只读静态 `publishView`。
