# 视频号发布面板半步迁移设计稿

- 日期：2026-04-05
- 主题：WeChat Publish Panel Migration
- 项目：`next-shadcn-admin-dashboard`
- 目标：在已经落地的“统一商品发布页”框架下，把 `视频号` 扩展区从纯 section metadata 升级成真实可编辑发布面板，同时保持详情页、首页、工作台与共享 store 的联动不变。

## 1. 背景

当前项目已经完成两层基础：

1. 商品详情页仍然承载主档摘要、渠道状态、字段编辑和动作入口
2. 新增了独立发布页 `/dashboard/e-commerce/products/[productId]/publish`
3. 发布页已经形成 `通用商品区 + 平台配置区 + 发布校验区`
4. 视频号扩展区已经具备 6 段 `extensionSections` 元数据
5. 抖店扩展区已经具备类目属性块和规则块的 mock 骨架

但当前新发布页仍有一个明显缺口：

- `视频号` 在发布页中还只是“结构存在”
- 真正能编辑、能保存、能联动的能力仍然留在详情页里的 `ChannelFieldEditor`

这会导致系统语义不一致：

1. 架构上已经说“真正的发布编辑工作台在独立发布页”
2. 但实际可用的深度编辑能力还停留在详情页

因此需要做一次“半步迁移”：

- 不一次性删除详情页旧编辑器
- 但先让新发布页里的 `视频号` 成为真实可编辑面板

## 2. 目标与范围

### 2.1 目标

本阶段目标是：

1. 让新发布页中的 `视频号` Tab 具备真实编辑和保存能力
2. 让发布页里的视频号区更像“发品编辑面”，而不是 section metadata 列表
3. 继续复用现有 store、保存链路、联动机制
4. 同时把详情页的角色逐步降级为“摘要 + 快捷入口”

### 2.2 范围

本阶段只迁移：

1. 视频号平台状态摘要
2. 视频号真实字段编辑与保存
3. 视频号缺失字段 / 发品缺口提醒
4. 发布页与详情页之间的职责再平衡

### 2.3 非目标

本阶段明确不做：

1. 抖店真实编辑能力迁移
2. 视频号真实 API 接入
3. 新建第二套发布草稿 store
4. 删除详情页旧编辑器
5. 把异常动作、上架/下架动作一并迁到新发布页

## 3. 已确认决策

### 3.1 迁移深度

本阶段采用“半步迁移”：

- 新发布页先接 `视频号` 的真实编辑能力
- 详情页保留旧编辑入口，但语义降级为过渡期能力

### 3.2 发布页承载内容

新发布页里的 `视频号` 扩展区，本阶段只承载：

1. 平台状态摘要
2. 字段编辑
3. 保存联动

不承载：

1. 重新同步
2. 上架
3. 下架
4. 其它动作链

### 3.3 技术方向

选择的方案是：

- 复用现有 `ChannelFieldEditor + WechatSkuPublishingSection + saveDraft`
- 在新发布页里增加一层发布语义包装，而不是重写一套新编辑器

## 4. 页面信息架构

## 4.1 新发布页中的视频号区

发布页里的 `视频号 Tab` 调整为三段结构：

### 4.1.1 平台状态摘要

放在 Tab 顶部，展示：

- 当前发布状态
- 审核状态
- 上下架状态
- 缺失字段数
- 最近同步时间

这段回答的问题是：

- 当前视频号发品卡在哪一步
- 当前是否适合继续补字段或进入后续提交流程

### 4.1.2 视频号最小可发品表单

中间主体直接复用现有真实编辑能力，但在发布页里按发品语义呈现：

1. 发品基础
2. 类目与品牌
3. 主图与详情
4. 履约与发货
5. SKU 发售信息
6. 参数 / 资质 / 服务

这里不再强调“渠道字段编辑器”的通用语义，而改为“视频号发品编辑”。

### 4.1.3 平台缺失与校验提醒

放在表单后或表单侧边，轻量展示：

- 当前缺哪些字段
- 保存后状态会如何变化
- 当前还不能进入哪些后续动作

这一段是发品编辑的补充语义，不再重复全页右侧的通用发布校验卡。

## 4.2 详情页角色调整

详情页本阶段不硬删旧能力，但语义要收口：

1. 详情页继续展示视频号状态和快捷处理入口
2. 详情页继续保留旧编辑能力，作为过渡期 fallback
3. 详情页文案要明确提示：
   - 深度发品编辑请前往发布页
   - 当前页更适合摘要查看和快速处理

这意味着详情页的角色将逐步转向：

- `主档摘要 + 渠道状态 + 快速处理入口`

而不是长期承担完整发布编辑工作台。

## 5. 组件与复用策略

### 5.1 直接复用

本阶段直接复用：

1. `ChannelFieldEditor`
2. `WechatSkuPublishingSection`
3. `product-center-editing.config.ts`
4. `product-center-editing.ts`
5. `saveDraft` 与共享 store 写链路

这些能力已经具备：

- schema 分组
- 表单校验
- SKU 发售信息编辑
- 保存回写
- 首页 / 详情页 / 工作台联动

### 5.2 新增包装层

本阶段只增加发布页语义包装组件，不重写底层编辑器：

1. `WechatPublishStatusSummary`
   - 发布页里的视频号状态摘要卡

2. `WechatPublishPanel`
   - 组合 `状态摘要 + 编辑器 + 缺失提醒`
   - 作为发布页里的真实视频号扩展区

3. `PlatformPublishPanel`
   - 从“只渲染 section metadata”升级成：
     - `wechat` 渲染真实可编辑 panel
     - `douyin` 继续渲染 mock-driven 骨架

这一步的核心不是重做编辑器，而是：

- `旧能力复用`
- `新页面语义承接`

## 6. 数据流与状态边界

### 6.1 数据源

新发布页里的视频号面板继续读取同一份 `product-center store`：

- 商品详情页和发布页不拆成两套商品状态
- 发布页直接取 `product.channels.wechat`

### 6.2 保存链路

发布页里的视频号保存继续走现有 `saveDraft(productId, "wechat", draft)`。

这样保存后会自然联动：

1. 发布页自身状态摘要
2. 详情页渠道状态
3. 商品中心首页
4. 发布工作台

### 6.3 明确不新建草稿状态层

本阶段不新增：

- `publish draft store`
- `publish route local cache`
- `独立于详情页的第二套视频号状态源`

原因是这会立即造成：

1. 发布页一份状态
2. 详情页一份状态
3. 首页 / 工作台再读一份投影视图

从而破坏当前已经建立好的共享联动关系。

## 7. 详情页降级策略

这轮不是立即移除详情页视频号编辑器，而是做“软降级”：

### 7.1 保留

保留：

1. 状态摘要
2. 异常提示
3. 快捷处理入口
4. 旧编辑能力

### 7.2 调整

调整：

1. 文案上不再把详情页当成主编辑面
2. 更明确地把发布页标成深度编辑入口

### 7.3 后续演进

后续如果发布页视频号编辑稳定，再进一步考虑：

1. 详情页仅保留只读摘要
2. 详情页只保留少量快捷处理动作
3. 深编辑完全迁到发布页

## 8. 验收标准

本阶段完成标准：

1. 新发布页中的 `视频号` Tab 已经是可编辑、可保存的真实面板
2. 发布页中的视频号区看起来像“发品编辑面”，不再只是 section metadata
3. 保存后继续联动：
   - 发布页
   - 详情页
   - 首页
   - 工作台
4. 详情页仍然可用，但语义已下调为“摘要 + 过渡入口”
5. 抖店扩展区不受影响，仍保持 mock-driven 骨架

## 9. 代码改造边界

建议优先改动这些区域：

1. 发布页平台面板相关组件
2. 发布页视频号专属包装组件
3. 详情页视频号文案与入口语义

建议本阶段避免：

1. 改写 `product-center-editing` 的底层数据模型
2. 新建第二套表单引擎
3. 把抖店和视频号同时做成真实可编辑迁移

## 10. 结论

这一步的本质不是“再做一个视频号编辑器”，而是：

- 把现有已成熟的视频号真实编辑能力
- 从详情页局部能力
- 提升为新发布页中的第一块真实平台扩展面板

这样做的收益是：

1. 独立发布页开始真正承担“发布编辑工作台”的职责
2. 不破坏现有共享 store 和跨页面联动
3. 为后续抖店真实编辑迁移保留一条清晰路径

这是统一发布架构从“结构成立”走向“能力真正迁移”的第一步。
