# 商品详情素材预览 Mock 设计稿

- 日期：2026-04-03
- 主题：Product Detail Asset Preview Mock
- 项目：`next-shadcn-admin-dashboard`
- 目标：把商品详情页右侧素材区从“素材状态列表”升级成可直观看布局效果的预览区，用一组更完整的 mock 数据验证封面图、卖点组图、详情长图在详情页中的排布是否成立。

## 1. 背景

当前商品详情页右侧已有素材区，但表现仍然停留在“素材名称 + 状态”层面：

1. 看得出素材是否齐备。
2. 看不出不同图片类型的布局差异。
3. 无法判断封面图、组图、长图在实际详情页里是否顺眼。

这会直接影响后续两个方向的判断：

1. 商品详情页右侧是否适合承担素材预览职责。
2. 未来素材模型是否需要支持更细的图片组织方式。

本次工作不解决素材管理，不接真实图片服务，只做一轮高保真 mock 预览验证。

## 2. 已确认决策

### 2.1 验证目标

- 重点验证“图片布局效果”，不是验证上传、裁剪或素材管理流程。
- 优先看封面图、组图、详情长图三类素材在详情页中的视觉节奏。

### 2.2 目标商品

- 第一轮只覆盖 `冷萃挂耳咖啡礼盒`。
- 原因是它最适合拉开三类素材的布局差异：
  - 封面主视觉
  - 多张卖点组图
  - 分段详情长图

### 2.3 展示位置

- 仅改商品详情页右侧素材区。
- 不改整个详情页信息架构。
- 不把后台页面直接改造成完整 C 端详情页。

### 2.4 展示方式

采用“详情页模拟式”预览，而不是纯后台素材工作台式列表。

也就是说，右侧素材卡不只展示素材是否存在，而是直接模拟用户浏览详情图文的节奏：

1. 先看封面图
2. 再看卖点组图
3. 再看详情长图

### 2.5 范围边界

本次明确不做：

- 真实上传
- 独立素材管理页
- 真轮播
- 真 lightbox
- 全商品补图
- 左侧详情主内容重排

## 3. 目标界面

当前 [`assets-card.tsx`](src/app/(main)/dashboard/e-commerce/products/[productId]/_components/assets-card.tsx) 将被升级成“详情素材预览卡”。

卡片内部固定分成三段。

### 3.1 封面图区

位置：

- 卡片最上方

表现：

- 一张大图
- 视觉权重最高
- 保留轻量状态标识

职责：

- 验证首图是否足够突出
- 验证封面比例和后台卡片宽度是否匹配

### 3.2 卖点组图区

位置：

- 封面图区下方

表现：

- 2 到 4 张图片网格
- 图片尺寸统一，但允许文案信息层次不同
- 不做真轮播，只做网格节奏

职责：

- 验证组图排布是否凌乱
- 验证组图在后台详情页里是否需要更强的分组边界

### 3.3 详情长图区

位置：

- 卡片底部

表现：

- 2 到 3 段长图连续预览
- 每段作为一个独立切片
- 展示纵向详情页节奏，而不是只放一张抽象“长图已就绪”

职责：

- 验证长图信息密度是否适合在右侧窄栏中预览
- 判断未来是否要做折叠、分段或单独查看入口

## 4. Mock 数据设计

当前 `assets` 结构只有：

- `id`
- `type`
- `label`
- `status`

这不够支撑真实预览，需要补成“能驱动视觉”的结构。

### 4.1 建议新增字段

针对本次预览，素材项至少补以下字段：

- `previewUrl`
  - 预览图地址
- `width`
  - 原始宽度
- `height`
  - 原始高度
- `variant`
  - 细分素材角色，例如：
    - `cover-hero`
    - `gallery-card`
    - `detail-section`

如当前 `AssetItem` 不适合直接承载多张图，可在 mock 层引入：

- `previewGroups`
  - 用于组织封面图、组图、详情长图

重点不是最终字段名，而是要有能力区分三类布局。

### 4.2 组图组织方式

“卖点组图”不能再是一条抽象素材。

第一轮建议至少补成 3 张：

1. 礼盒主视觉
2. 风味卖点图
3. 冲泡/送礼场景图

### 4.3 长图组织方式

“详情长图”建议拆成 2 到 3 段：

1. 核心卖点段
2. 产品规格/冲泡说明段
3. 礼盒细节或场景段

这样前端才能真实验证“长图分段预览”是否成立。

## 5. 视觉方向

这次 mock 的关键不是素材真假，而是节奏真假。

### 5.1 素材风格

建议使用统一风格的 mock 图：

- 咖啡豆、挂耳袋、礼盒包装、杯具场景
- 综合色调偏暖咖、奶白、深棕
- 图中允许有轻量文案块与信息排版

避免：

- 随机来源的杂乱图片
- 风格不统一的占位图
- 只用纯色块代替全部图片

### 5.2 后台气质保持

虽然采用“详情页模拟式”预览，但页面仍然要保留后台产品的气质：

- 仍然是卡片容器
- 仍然有轻量状态标识
- 仍然服从当前右侧窄栏宽度

不应该看起来像一个完整的 C 端详情页嵌进后台。

## 6. 前端实现建议

### 6.1 数据层

推荐两种做法中的第一种：

1. 在现有 mock 数据中直接补预览字段，并在 `AssetItem` 上做最小扩展
2. 额外加一层 `assetPreviewMock` 映射表，避免直接改原始素材结构

推荐方案是 `2`：

- 可以降低对现有渠道编辑逻辑的干扰
- 当前渠道字段编辑仍然只依赖 `assets` 的基础信息
- 详情预览卡单独消费更丰富的 preview mock 数据

### 6.2 组件层

建议仅修改：

- `assets-card.tsx`

如组件体积明显变大，可拆成：

- `asset-preview-cover.tsx`
- `asset-preview-gallery.tsx`
- `asset-preview-detail-strip.tsx`

但第一轮可以先保守，优先在一个组件内验证布局。

### 6.3 状态层

本次不需要把素材预览接入 store 写操作。

它只是只读 mock 预览，因此读取静态 mock 即可。

## 7. 成功标准

本次完成后，应满足以下条件：

1. 打开 `冷萃挂耳咖啡礼盒` 详情页时，右侧素材区能直观看出三类图片布局差异。
2. 用户能明确区分封面图、卖点组图、详情长图各自承担的作用。
3. 页面整体仍然保持当前后台详情页风格，不突兀。
4. mock 数据结构足够清晰，后续可以继续扩展到更多商品或替换成真实素材。

## 8. 后续衔接

如果这轮布局验证成立，下一步可从以下方向中选一个继续推进：

1. 把同样的素材预览能力扩到第二个商品
2. 将右侧素材预览扩展成独立素材模块
3. 将当前 preview mock 抽象为后续本地 API/mock contract 的一部分

本次设计只覆盖第一步验证，不预支后续范围。
