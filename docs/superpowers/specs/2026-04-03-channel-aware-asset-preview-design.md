# 商品详情按渠道切换素材预览设计稿

- 日期：2026-04-03
- 主题：Channel-Aware Product Detail Asset Preview
- 项目：`next-shadcn-admin-dashboard`
- 目标：让商品详情页右侧素材预览跟随当前渠道 Tab 切换，在同一商品下直观看到 `抖音` 与 `视频号` 两套素材预览在视觉风格和素材编排上的差异。

## 1. 背景

上一轮已经把商品详情页右侧素材区从“素材状态列表”升级为三段式预览：

1. 封面图
2. 卖点组图
3. 详情长图

这解决了“能不能看布局”的问题，但仍然只是一套与渠道无关的 preview mock。

对于当前商品中心的目标，这还不够，因为：

1. 页面左侧已经有 `抖音 / 视频号` 渠道 Tab。
2. 右侧素材预览如果不跟着渠道切，会默认暗示“各渠道素材展示策略完全一致”。
3. 你当前要验证的重点，已经从“素材区能不能看图”推进到了“同一商品在不同渠道下怎么摆图更顺”。

因此这一轮的核心不是新增页面，而是把已有预览能力升级为“按渠道切换”。

## 2. 已确认决策

### 2.1 交互方式

采用“跟随当前渠道 Tab 切换”的方式：

1. 用户在详情页左侧切换 `抖音 / 视频号`
2. 右侧素材预览同步切换到对应渠道版本

明确不采用：

- 右侧双列同时对比
- 右侧独立渠道切换器

原因：

- 现有详情页结构最适合单一主视角
- 避免右侧栏因多套控制器变得拥挤
- 保持用户心智一致：左侧在看哪个渠道，右侧就看哪个渠道的素材预览

### 2.2 差异层级

本轮差异不是只换颜色，也不是完全独立两套系统，而是：

- 视觉风格差异
- 素材编排差异

也就是在同一商品下，`抖音` 与 `视频号` 仍然共用三段预览结构：

1. 封面图
2. 卖点组图
3. 详情长图

但三段里的文案语气、信息密度、素材重心需要明显不同。

### 2.3 范围边界

本轮明确不做：

- 新增第三个平台
- 真实平台素材接口
- 独立素材设置页
- 右侧双列比较模式
- 素材上传、选择器或编辑动作

## 3. 目标界面

### 3.1 页面联动关系

商品详情页的主结构不变：

- 左侧：商品基础信息 + 渠道发布 Tab
- 右侧：SKU + 素材预览

唯一变化是右侧素材预览不再只看 `productId`，而要同时看：

- `productId`
- `activeChannel`

### 3.2 切换行为

当用户切换 Tab 时：

1. 左侧渠道状态内容变化
2. 右侧 `AssetsCard` 内的三段预览同步变化

右侧不额外出现第二个渠道选择控件。

### 3.3 视觉效果预期

切换到 `抖音` 时，右侧应该更像“强主视觉 + 快节奏卖点卡 + 更直接的详情段”。

切换到 `视频号` 时，右侧应该更像“内容场景封面 + 生活方式组图 + 更连续的长图阅读节奏”。

用户不需要看字段差异，仅通过视觉就能理解：

- 抖音更偏快速抓取注意力和转化
- 视频号更偏内容场景和阅读感

## 4. Mock 数据设计

### 4.1 数据组织方式

当前 preview mock 是：

- `productId -> preview`

本轮升级为：

- `productId -> channel -> preview`

也就是类似：

- `spu-hanger-coffee.douyin`
- `spu-hanger-coffee.wechat`

推荐结构：

```ts
type ChannelAssetPreviewMap = Partial<Record<ChannelId, ProductAssetPreview>>;
type ProductAssetPreviewMap = Partial<Record<string, ChannelAssetPreviewMap>>;
```

重点不是字面类型名，而是要把“商品维度”和“渠道维度”拆清楚。

### 4.2 读取接口

推荐把读取 helper 从：

```ts
getProductAssetPreview(productId)
```

升级成：

```ts
getProductAssetPreview(productId, channelId)
```

返回值仍然是：

- `ProductAssetPreview | undefined`

如果该商品或该渠道未配置 preview mock，则返回 `undefined`。

## 5. 渠道差异设计

### 5.1 抖音版 preview

整体方向：

- 更强主视觉
- 更快信息节奏
- 更偏转化导向

建议表现：

- `封面图`
  - 标题更大
  - 对比更强
  - 更突出礼盒、爆款、送礼感
- `卖点组图`
  - 更像卖点卡
  - 强调风味、礼盒结构、场景利益点
- `详情长图`
  - 每段信息更短、更直接
  - 更像“快速浏览后可以成交”的详情节奏

### 5.2 视频号版 preview

整体方向：

- 更内容化
- 更克制
- 更偏场景阅读感

建议表现：

- `封面图`
  - 更像生活方式封面
  - 降低强促销感
- `卖点组图`
  - 更强调饮用场景、桌面氛围、内容感
  - 不只是“卖点堆叠”
- `详情长图`
  - 更连续
  - 阅读感更强
  - 像内容详情页，而不是纯货架说明图

### 5.3 差异控制原则

这轮差异只落在三类层面：

1. 文案语气
2. 配色和视觉强弱
3. 信息密度和素材编排

明确不做：

- 完全不同字段结构
- 完全不同组件结构
- 不同渠道使用不同页面骨架

## 6. 前端实现建议

### 6.1 数据层

优先保留当前独立 preview mock 方案，不回写到 `AssetItem`。

原因：

1. 现有 `assets` 仍然服务“真实素材状态”语义
2. 当前 preview 只是 layout mock
3. 如果把渠道 preview 直接塞进 `assets`，会和当前详情页里的素材状态心智打架

### 6.2 组件层

最小变化建议：

1. `ProductDetailPage` 持有当前激活渠道状态
2. `ChannelPublicationTabs` 在 tab 切换时把 `activeChannel` 往外通知
3. `AssetsCard` 增加 `channelId` 入参
4. `AssetsCard` 根据 `productId + channelId` 读取 preview

如果当前 `ChannelPublicationTabs` 不方便直接暴露 state，可新增轻量回调：

```ts
onChannelChange?: (channelId: ChannelId) => void
```

### 6.3 预览组件

现有三段预览组件可以继续复用：

- `AssetPreviewCover`
- `AssetPreviewGallery`
- `AssetPreviewDetailStrip`

本轮重点不在重写布局组件，而在：

1. preview 数据切换
2. 不同渠道 mock 的视觉差异
3. 页面联动是否顺畅

## 7. 成功标准

本次完成后，应满足以下条件：

1. 在商品详情页切换 `抖音 / 视频号` 时，右侧素材预览同步切换。
2. 两套 preview 一眼能看出视觉风格和素材编排差异。
3. 页面整体仍然保持当前后台结构，不新增复杂控制器。
4. 数据结构已经自然指向未来的“渠道化素材预览 / 映射”能力。

## 8. 后续衔接

如果这轮按渠道切换验证成立，下一步可从以下方向中选择：

1. 把第二个商品也补成双渠道 preview
2. 把 preview mock 与渠道字段编辑联动
3. 将当前 preview helper 抽成本地 API/mock contract 的一部分

本次设计只覆盖“同一商品下的双渠道素材预览切换”，不预支后续接口与编辑能力。
