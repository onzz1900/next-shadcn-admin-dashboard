# 全渠道商品中心设计稿

- 日期：2026-04-02
- 主题：Omni-Channel Product Center
- 项目：`next-shadcn-admin-dashboard`
- 目标：基于当前项目现有的后台框架风格，先构建一个可演进的前端骨架，用页面信息架构反向逼出商品中心主档、渠道适配、发布动作三层数据边界。

## 1. 背景与目标

当前项目是基于 Next.js App Router、shadcn/ui、TanStack Table 的后台模板，适合承载“摘要卡 + 筛选区 + 数据表格 + 详情分区”的业务页面。本次不直接从后端领域建模起步，而是从真实业务页面倒推系统结构。

第一阶段的核心目标：

1. 设计一个平台无关的商品中心前端框架。
2. 以抖音、视频号作为首批样板平台验证渠道差异。
3. 以 `SPU` 为商品主档核心，`SKU` 作为销售规格附属层。
4. 同时支持“商品主档视角”和“渠道发布视角”，但默认首页突出商品主档。
5. 产出可演进的前端骨架，而不是一次性的静态原型。

## 2. 已确认决策

### 2.1 业务范围

- 第一阶段做平台无关的商品中心。
- 首批渠道样板为 `抖音`、`视频号`。
- 页面同时承载商品主档和渠道发布两个视角。
- 默认首页突出 `商品主档` 视角。
- 商品主数据以 `SPU` 为核心，`SKU` 作为规格附属层。

### 2.2 推荐方案

采用“商品主档优先，渠道发布为从属视图”的方案。

原因：

- 商品中心的长期复用能力依赖于平台无关的主档模型。
- 抖音、视频号的渠道差异更适合通过适配层挂载到主档，而不是反向主导整体信息架构。
- 当前项目现有组件和页面模式更适合“主档列表 + 详情分区 + 渠道子视图”，不适合第一版就做高密度运营控制台。

## 3. 第一阶段页面信息架构

第一阶段建议拆成 4 个页面/视图层级。

### 3.1 商品中心首页

默认入口，围绕 `SPU` 主档列表展开。

核心问题：

- 有哪些商品。
- 哪些商品具备发布条件。
- 哪些商品卡在资料、审核、渠道状态上。

页面结构：

1. 业务摘要卡区
2. 筛选区
3. 商品列表区
4. 行级快捷操作区

### 3.2 商品详情页

以单个 `SPU` 为中心展示商品全貌，并承载主档维护与渠道发布的汇合操作。

页面结构：

1. 基础信息
2. 销售规格 `SKU`
3. 内容素材
4. 渠道发布

### 3.3 渠道发布子视图

渠道发布不独立成另一套商品系统，而是作为商品详情中的强操作区域存在。

建议按 `抖音`、`视频号` 分 Tab 或分段展示以下内容：

- 待补字段
- 审核状态
- 上下架状态
- 失败原因
- 最近同步记录
- 可执行动作

### 3.4 发布工作台

作为辅助页存在，用于集中处理运营任务，但不作为默认首页。

建议承载的聚合视角：

- 待审核
- 审核失败
- 待上架
- 同步异常

## 4. 首页与详情页布局设计

### 4.1 商品中心首页布局

首页建议采用“上摘要、下列表”的后台工作页结构。

顶部摘要卡建议展示：

- 商品总数
- 待补资料
- 可发布
- 审核中
- 发布异常

筛选区建议至少包含：

- 商品状态
- 渠道状态
- 类目
- 品牌或店铺
- 更新时间
- 关键词

商品列表建议包含以下列：

- 主图
- 商品名称 / SPU 编码
- 类目
- SKU 数
- 资料完整度
- 抖音状态
- 视频号状态
- 最后更新时间
- 快捷操作

设计原则：

- 重点不是平铺所有字段，而是让用户快速判断是否可发布、卡在哪个平台。
- 列表应优先承载“决策信息”，详细字段放到详情页。

### 4.2 商品详情页布局

详情页不建议做成长表单，而应分区组织。

建议区块：

1. 基础信息区
2. SKU 区
3. 素材区
4. 渠道发布区

渠道发布区按平台展示统一结构：

- 发布所需字段映射
- 当前状态
- 驳回或失败原因
- 最近同步结果
- 操作按钮：提交审核、更新、上架、下架、重新同步

设计原则：

- 用户应能在同一页面完成“看主档、补资料、看平台差异、执行发布动作”的完整闭环。
- 渠道差异通过结构化区块承载，不通过整页复制两套布局解决。

## 5. 前端模块边界

第一阶段前端建议先拆出 5 个清晰模块。

### 5.1 商品主档列表模块

职责：

- 首页摘要卡
- 列表筛选
- 分页与排序
- 行级状态展示

输入模型：

- `SPUSummary`

目标：

- 输出平台无关的商品总览，不直接暴露抖音或视频号原始字段。

### 5.2 商品详情主档模块

职责：

- 基础信息展示
- 类目、品牌、属性
- 图文素材
- SKU 结构

输入模型：

- `SPUDetail`

目标：

- 围绕页面所需信息组织主档，而不是照搬平台接口。

### 5.3 渠道适配模块

职责：

- 渠道发布状态展示
- 待补字段归集
- 平台特有字段承载
- 最近回执显示

输入模型：

- `ChannelPublicationView`

目标：

- 用统一视图模型包住抖音、视频号差异。

### 5.4 发布动作模块

职责：

- 提交审核
- 更新渠道商品
- 上架
- 下架
- 重新同步

输入/输出模型：

- `ChannelActionRequest`
- `ChannelActionResult`

目标：

- 第一阶段即使先用假动作，也要把动作契约和反馈结构固定下来。

### 5.5 任务与回执模块

职责：

- 同步结果记录
- 审核驳回原因
- 字段校验失败信息
- 最近处理时间

输入模型：

- `PublicationTaskSummary`

目标：

- 为未来独立任务中心预留清晰边界。

## 6. 由页面倒推的数据边界

页面不是简单“向后端取数”，而是反向逼出三类系统能力。

### 6.1 商品中心主数据

服务对象：

- 首页列表
- 商品详情主档
- SKU 与素材区

建议抽象：

- `SPUSummary`
- `SPUDetail`
- `SKUItem`
- `AssetItem`

### 6.2 渠道适配结果

服务对象：

- 渠道发布区
- 首页渠道状态列

建议抽象：

- `ChannelPublicationView`

建议字段方向：

- `channel`
- `publicationStatus`
- `auditStatus`
- `listingStatus`
- `missingFields`
- `channelSpecificFields`
- `lastSyncAt`
- `lastResult`

### 6.3 渠道动作回执与任务状态

服务对象：

- 按钮动作反馈
- 失败原因展示
- 发布工作台

建议抽象：

- `ChannelActionResult`
- `PublicationTaskSummary`

这三层边界确定后，后续后端能力可以按页面真实需求逐层补齐，而不是一开始被平台 API 细节牵着走。

## 7. 关键交互流

第一阶段至少要覆盖以下用户流：

1. 用户进入首页，看到商品总览和渠道风险摘要。
2. 用户通过筛选定位到某个需要发布的商品。
3. 用户进入商品详情页，查看主档完整度和 SKU、素材情况。
4. 用户切换到抖音或视频号发布区，看到待补字段和当前状态。
5. 用户触发提交审核、更新或上架等动作，并收到统一结构的反馈。

## 8. 错误处理与状态表达

第一阶段即使不接真实 API，也应提前统一状态表达方式。

### 8.1 页面状态

- 加载中
- 空数据
- 可操作
- 部分缺失
- 动作执行中
- 动作失败

### 8.2 渠道状态

- 未发布
- 待补资料
- 待审核
- 审核中
- 审核失败
- 待上架
- 已上架
- 已下架
- 同步异常

### 8.3 错误信息原则

- 优先展示对运营可行动的错误，而不是技术错误原文。
- 驳回或失败原因应直接关联到字段或动作。
- 平台特定错误应在统一结构下保留原始语义。

## 9. 第一阶段明确范围

### 9.1 In Scope

- 商品中心首页
- 商品详情页
- 详情页内的渠道发布子视图
- 抖音、视频号两个渠道样板
- 平台无关主档结构
- 统一的渠道状态和动作反馈框架

### 9.2 Out of Scope

- 真实平台 API 接入
- 真实审核、上架、同步动作
- 批量发布
- 独立任务中心页面
- 多平台扩展到三个以上
- 复杂权限控制

## 10. 验证标准

第一阶段完成后，应满足以下标准：

1. 用户能在首页判断商品是否具备发布条件。
2. 用户能在详情页明确区分主档信息与渠道差异。
3. 抖音、视频号的状态、缺失字段、失败原因能通过统一结构展示。
4. 页面结构已经足以反向定义后续主档接口、渠道适配接口、动作接口。

## 11. 实现前建议

在真正进入实现前，下一步应先写一份实现计划，至少覆盖以下内容：

1. 路由与页面文件组织方式
2. mock 数据模型与类型定义
3. 首页表格与详情页分区的组件拆分
4. 渠道发布子视图的数据适配层
5. 最小可验证页面路径与检查项

## 12. 风险与后续演进

### 12.1 当前主要风险

- 若首页同时承担过多渠道运营职责，容易压垮主档视角。
- 若直接照抄抖音或视频号字段，通用商品中心会失去中台价值。
- 若第一阶段不先统一动作反馈结构，后面接入真实接口时会出现页面与能力错位。

### 12.2 后续演进方向

- 增加第三个平台时复用现有 `ChannelPublicationView`
- 将发布工作台从辅助页升级为任务中心
- 补齐批量动作、权限、异步任务、审计日志等能力

---

结论：第一阶段应先落成一个以 `SPU` 为锚点的商品主档中心，再通过统一渠道发布子视图包住抖音、视频号差异。页面先行，但页面必须反向收敛成明确的数据能力边界，才能支撑后续真实平台接入。
